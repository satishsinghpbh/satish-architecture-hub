Critical Implementation Traps — Must Get Right
② Spring AOP Self-Invocation
⚠ onMessage() calls processFile() in SAME class → Spring @Transactional proxy is completely bypassed → no transaction.
processFile() must live in a separate @Service (TradeFileProcessorService) injected into the listener.
③ fetchWithRetry Isolation Trap
⚠ If inside @Transactional, REPEATABLE_READ caches the first empty result. Every retry returns the same empty result forever.
Call fetchWithRetry() BEFORE any @Transactional boundary. Each attempt is a fresh independent read.
④ markInProgress Propagation Trap
⚠ If markInProgress uses REQUIRES_NEW it commits independently. App crash mid-row → status stuck IN_PROGRESS forever.
Use default REQUIRES propagation → joins parent tx → rolls back together on failure → status reverts to SUBMITTED.

Why REQUIRES_NEW? The DLQ insert must succeed even if the listener's outer transaction is in a broken state. Sharing the transaction would cause DLQ save to also roll back — the message record would be silently lost.
⑤ IBM MQ Backout Queue — Permanent Failure Path
⚠ Corrupt file → parseCsv() always throws → listener nacks → IBM MQ redelivers → infinite loop. Without backout configured, queue is stuck forever.
Configure IBM MQ BackoutThreshold=5. After 5 failed redeliveries, IBM MQ auto-moves message to TRADE.FILE.BACKOUT queue. A BackoutQueueListener sets status=FAILED and inserts to trade_file_dlq.

IBM MQ CLIENT_ACKNOWLEDGE — Key Behaviours
Bulk-ACK Risk
⚠ JMS spec: calling acknowledge() on message N acknowledges ALL messages received in that session up to N. Risk if processing multiple concurrently.
Set maxMessagesPerTask=1. One message per session. ACKs exactly that one message.
factory.setMaxMessagesPerTask(1);
factory.setSessionAcknowledgeMode(Session.CLIENT_ACKNOWLEDGE);
MQ Session Timeout During Long Processing
⚠ If file processing exceeds IBM MQ HBINT (heartbeat) timeout, session is dropped → IBM MQ redelivers to another instance while first is still running.
Two instances race → atomic UPDATE WHERE SUBMITTED → one wins. UNIQUE(file_id, row_num) handles any partial row overlap.
Set IBM MQ HBINT > max expected processing time. Or chunk large files to keep processing fast.
Large File Handling — Chunked Processing
⚠ 100,000 row file in single @Transactional → long-running tx → DB connection timeout, lock escalation, OOM loading all rows into memory at once.
Chunk rows into batches of 500. Each batch in its own @Transactional(REQUIRES_NEW). Track last_processed_row as checkpoint in trade_file. On redelivery → skip already-committed batches.
ALTER TABLE trade_file ADD COLUMN last_processed_row INT DEFAULT 0;
-- On each batch commit → UPDATE last_processed_row = batchEndRow
-- On redelivery → resume from last_processed_row + 1
-- UNIQUE(file_id, row_number) still protects against any overlap
